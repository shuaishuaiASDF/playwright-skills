# Playwright Matrix Plan

## 作用

使用本 Skill 生成 Playwright 测试矩阵落地方案。它只负责分析、梳理、映射和规划，不默认改测试脚本；只有用户明确要求继续落地时，才进入代码实现。

目标产物是一个稳定模板化的方案，而不是某次执行流水。方案必须回答：

1. 当前模块应该测什么。
2. 现有文档、测试用例和 Playwright 脚本已经覆盖什么。
3. 哪些缺口需要新增、修改或补强 Playwright 用例。
4. 后续应按什么公共能力和批次落地。

## 使用前提

若用户只要求方案，先输出方案，不修改文档或代码。

## 模式判断

先根据用户输入判断模式，并在输出开头写明本次模式：

| 模式 | 触发条件 | 输出重点 |
|---|---|---|
| 矩阵-only 模式 | 用户只要求某模块测试矩阵、覆盖缺口、落地方案，未提供外部测试用例 | 测试矩阵、覆盖缺口、Playwright 新增/修改建议、公共能力、批次计划 |
| 测试用例映射模式 | 用户提供 Excel、Markdown、表格、文本测试用例，主要要求归并或映射 | 测试用例到矩阵场景的主映射、辅助映射、映射理由、重复/冲突处理 |
| 组合模式 | 用户既指定模块现状，又提供外部测试用例 | 完整输出矩阵、测试用例映射、证据口径、Playwright 落地拆解、公共能力、批次计划 |
| 增量更新模式 | 用户已有矩阵方案，本次只更新部分场景、补充新用例或修正覆盖状态 | 只更新变更部分，保留已确认内容不变，每处变更标注变更原因；不重写整篇方案 |

如果模式不清楚，按组合模式分析，但不要编造不存在的测试用例来源。

### 模式判断边界示例

| 用户输入示例 | 判定模式 | 依据 |
|---|---|---|
| "帮我看看凭证模块应该测什么" | 矩阵-only | 未提供外部用例，只要求模块测试矩阵 |
| "这是一份 Excel 测试用例，帮我整理归并" | 测试用例映射 | 提供了外部用例，主要要求映射 |
| "凭证模块现状梳理，这里还有一份用例，一起出方案" | 组合模式 | 既有模块现状分析，又有外部用例 |
| "上一版方案补一下新增的几个场景" | 增量更新 | 已有方案，本次只更新部分内容 |

如果仍无法判断，默认组合模式。

## 最小阅读顺序

1. 读取项目规则：`.codex/rules/gdmcb-ai-code-rules.md`。
2. 读取目标模块已有说明文档，例如 `ui-test/docs/<模块>.md`。
3. 如果存在测试用例整理文档，读取 `ui-test/docs/excel/<模块>用例分类整理.md` 或用户提供的用例材料。
4. 读取目标模块已有 Playwright spec。
5. 读取必要的前端页面、组件、API 文件，用来确认页面真实能力和接口边界。
6. 需要判断报告、mock/real 或 DB 口径时，读取仓库内 `playwright-common` skill 或现有同类 spec/report 实现。

只读取当前任务需要的最小文件集。不要为了生成矩阵扫描无关模块。

## 输出流程

1. 先列输入来源和本次模式。
2. 识别模块边界、页面入口、角色、已有文档、已有 spec、外部测试用例来源。
3. 按业务目标归并场景，不按按钮或接口机械罗列。
4. 生成测试矩阵。
5. 若有外部测试用例，生成测试用例映射表。
6. 按需生成 DB、文件、跨角色、跨系统或 mock 辅助证据对照。
7. 拆解需要新增、需要修改、已覆盖但建议补强的 Playwright 用例。
8. 抽出最小公共能力。
9. 给出批次计划和完成标准。
10. 输出待确认项，禁止把未确认接口、字段、账号、样本、DB 表或业务规则写成既定事实。

## 固定规则

- 一个外部测试用例只能有一个主映射场景；它可以出现在其他场景的辅助/同类测试用例列，但不能被多次当作主证据。
- 覆盖状态必须由真实 Playwright 证据、可执行脚本、报告结果或明确代码证据支撑。
- `已覆盖` 只表示核心验证目标已被证明；只覆盖主链路但缺少关键分支、DB、文件、跨角色回查或真实样本时，标记为 `部分覆盖`。
- mock 优先用于前端分支、错误态、空态、按钮状态、loading、toast、弹窗关闭等可控页面行为。
- real 优先用于真实接口、真实入库、唯一性、文件解析、权限隔离、跨页面回查、跨角色回查和真实状态变化。
- DB、文件、跨系统证据只在业务目标明确需要时加入，不要为了显得完整而强行添加。
- 查询真实数据为 0 时，如实写 `查不到`，并按场景目标判定为 `FAIL`、`WARN`、`SKIP` 或保持 `部分覆盖`。
- 不要把 mock 结果当成真实业务结论；如果必须真实，mock 只能作为辅助证据。
- 不要重复计数同一场景；已有 mock 用例需要升级真实证据时，优先修改原 test，而不是新增同名真实 test。
- 批次计划必须区分已完成、待新增、待修改和待确认，不能把历史完成项继续留在待办里。

- 需要完整模板时，读取 `references/matrix-plan-template.md`。
- 需要判断映射、覆盖状态、mock/real/DB 证据边界时，读取 `references/mapping-rules.md`。

## 交付要求

最终回答必须说明：

1. 本次模式和输入来源。
2. 形成的矩阵或方案文件位置；如果只做口头方案，也说明未改文件。
3. 覆盖状态和落地建议的证据来源。
4. 是否涉及 DB、接口契约、测试数据、账号权限或跨系统联调。
5. 需要用户补充确认的事项。
