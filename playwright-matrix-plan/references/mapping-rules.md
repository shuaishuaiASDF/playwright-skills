# 测试矩阵映射与证据规则

## 1. 场景归并规则

1. 按主验证目标归并场景，不按外部用例行号、按钮、接口或页面区域机械归并。
2. 同一场景只能归入一种类型：正常流、边界流、异常流，必要时追加联调场景。
3. 正常流证明核心业务链路成立。
4. 边界流证明规则边界、状态边界、数量边界、格式边界成立。
5. 异常流证明失败处理、回滚、可重试、重复提交、接口异常、超时和错误提示可控。
6. 联调场景只在单页面无法证明业务闭环时使用，例如跨角色、跨页面、跨系统或上下游状态同步。

## 2. 外部测试用例映射规则

1. 每条外部测试用例只能有一个主映射场景。
2. 主映射选择最贴近该用例主验证目标的矩阵场景。
3. 如果一个外部用例同时涉及入口、操作、结果和下游回查，主映射放在最能证明最终业务目标的场景。
4. 如果一个外部用例只是为其他场景提供相似样本或边界补充，放入辅助/同类测试用例列。
5. 不要因为同一外部用例中出现多个页面动作，就把它拆成多个主映射。
6. 不要把多个主验证目标不同的外部用例硬合并成一个大场景。
7. 外部用例要求 DB、文件、跨角色或跨系统时，要在映射表外再补证据对照表。

## 3. 覆盖状态规则

| 状态 | 使用条件 | 禁止用法 |
|---|---|---|
| 已覆盖 | Playwright 已有用例证明该场景核心验证目标，断言覆盖 UI/API/DB/文件等必要证据 | 只有脚本存在但未断言核心业务结果时，不得标记 |
| 部分覆盖 | 主链路或局部分支已有证据，但外部用例细节、真实样本、DB、文件、跨角色或异常分支未补齐 | 不得把部分覆盖写成完成 |
| 未覆盖 | 场景已识别，但没有有效 Playwright 证据 | 不得因有手工用例而标记已覆盖 |
| 规划中 | 已明确落地方案或排期，但尚无自动化证据 | 不得代替真实执行结论 |
| 查不到 | 真实查询或真实样本返回 0，且这是业务观测结果 | 不得用 mock 命中覆盖真实查不到 |
| 待确认 | 依赖接口、字段、账号、样本、DB 表或业务规则未明确 | 不得编造后继续设计为已落地 |

### 覆盖状态升级路径

1. 覆盖状态只能由低向高单向升级：`未覆盖 → 部分覆盖 → 已覆盖`；不得跳级，也不得反向降级（除非发现原有证据有误）。
2. 从 `部分覆盖` 升级到 `已覆盖` 时，必须在方案中明确列出本次升级所补充的证据来源（Playwright 脚本标题、报告截图路径、DB 查询结果或文件校验结果）。
3. 升级不自动生效；方案中必须单独列出"本次覆盖状态变更"清单，写明每个场景的变更前状态、变更后状态和变更原因。
4. 若某场景依赖未满足，不得以"预计下批次补上"为由提前升级为 `已覆盖`；保持 `部分覆盖` 或 `规划中` 直到证据真正到位。

## 4. mock / real / DB / 文件选择规则

### 优先 real 的场景

- 登录、权限、主链路、保存、查询、查看、编辑、删除、导入、上传、下载后的真实业务结果。
- 唯一性、重复提交、幂等、状态变更、状态不变。
- 文件解析、导入模板、错误文件、上传后查询回看。
- 跨页面、跨角色、跨系统回查。
- 外部测试用例明确要求真实数据、DB 或业务最终状态。

### 优先 mock 的场景

- 接口 5xx、超时、队列满、任务创建失败等不可稳定制造的异常。
- 无数据、空态、按钮禁用、未勾选提示、弹窗取消、loading、toast、前端字段映射。
- 罕见状态组合或危险数据状态。
- 页面前端交互保护，不涉及真实持久化结论。

### 必须 DB 的场景

- 外部测试用例明确写了查询数据库。
- 页面或接口无法直接证明最终状态。
- 需要证明状态变化或状态不变。
- 需要证明重复提交未重复落库。
- 需要证明真实权限隔离或数据归属。

### 必须文件证据的场景

- 模板下载和模板列要求。
- 导入、上传、错误文件、下载、导出。
- 文件名、后缀、内容、错误行、错误原因、解析结果。

## 5. Playwright 用例拆解规则

1. 优先复用原 spec；只有当文件规模、执行耗时、环境依赖、报告隔离或风险边界确实需要时，才新增独立 spec。
2. 现有用例能承接的场景，优先修改现有 test。
3. 新增场景与既有场景目标不同，才新增 test。
4. 已有 mock 用例如果被判定必须真实，不新增同名 real test；应升级原 test 或明确拆分 mock 辅助和 real 主验证。
5. 每个建议新增/修改用例必须写清数据来源、核心断言和报告证据。
6. 新增 spec、公共能力或测试脚本后，应同步检查 `ui-test/README.md`。

## 6. 公共能力抽取规则

1. 三条以上用例复用或跨批次复用的逻辑，优先抽成 helper。
2. 只服务单条用例且业务变化大的逻辑，先放在 spec 内。
3. 即使复用未达三条，但实现复杂度高、失败排查成本高或涉及多步异步等待的逻辑（如 DB 回查链路、文件解析校验链路），也可以考虑抽成 helper，以降低单条用例的维护负担。
4. 公共能力命名按模块归属，不做跨模块大一统。
5. 常见公共能力包括：
   - 测试数据工厂
   - 页面进入和主链路 helper
   - 请求/响应等待 helper
   - 唯一键查询和回查 helper
   - 字段审计 helper
   - 文件上传/下载/导入/解析 helper
   - DB 查询和断言 helper
   - 跨角色或跨系统入口探测 helper

## 7. 批次排序规则

1. 阶段0先确认环境、账号、权限、样本、DB、文件和公共能力。
2. 先做低风险、高价值、低依赖场景。
3. 再做真实数据、文件、DB 依赖较强的场景。
4. 最后做跨角色、跨系统、外部系统和复杂异常。
5. 已完成项必须从待办批次移出，避免计划文档失真。
6. 被环境阻断的场景写清阻断条件，不升级覆盖状态。
7. 若某场景前置依赖未满足，不阻塞同批次其他独立场景继续落地；阻塞场景标记阻断条件后移至下一批次，覆盖状态保持不变。

## 8. 输出防错清单

交付前检查：

1. 是否写明本次模式。
2. 是否列出输入来源。
3. 是否把外部测试用例的主映射和辅助映射分开。
4. 是否每个覆盖状态都有证据或缺口说明。
5. 是否把真实必验场景错误地交给 mock 结案。
6. 是否把查不到伪装成已覆盖。
7. 是否把未确认接口、字段、DB 表、账号、样本写成事实。
8. 是否把已完成内容继续留在待办批次。
9. 是否说明是否涉及 DB、接口契约、测试数据、账号权限或跨系统联调。
