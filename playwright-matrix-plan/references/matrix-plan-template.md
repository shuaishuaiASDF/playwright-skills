# Playwright 测试矩阵落地方案模板

## 使用说明

本模板用于生成稳定的 Playwright 测试矩阵落地方案。根据输入选择三种模式之一：

| 模式 | 适用场景 | 必出章节 |
|---|---|---|
| 矩阵-only 模式 | 只梳理模块测试矩阵和 Playwright 落地方案 | 1、2、3、4、7、8、9、10、11 |
| 测试用例映射模式 | 已有外部测试用例，需要归并到矩阵 | 1、2、3、4、5、10、11 |
| 组合模式 | 同时分析模块现状和外部测试用例 | 全部章节 |

如果某章不适用，明确写“不适用”和原因，不要删除关键判断。

## 1. 文档头部

# <模块名称>测试矩阵落地方案

> 模块：`<模块路径或业务模块>`
>
> 页面/路由：`<页面路由，可多个>`
>
> 角色：`<测试角色或账号类型>`
>
> 本次模式：`矩阵-only / 测试用例映射 / 组合模式`
>
> 输入来源：
> - `<已有模块文档路径或说明>`
> - `<外部测试用例来源；没有则写无>`
> - `<已有 Playwright spec 路径；没有则写无>`
> - `<必要前端代码或 API 文件路径>`
>
> 输出目标：形成测试矩阵、覆盖缺口、测试用例映射和 Playwright 分批落地方案。
>
> 说明：本方案用于指导 Playwright 测试设计和落地；除非用户明确要求，不替换、不修改现有文档和脚本。

## 2. 编写原则

1. 矩阵按业务验证目标组织，不按页面按钮、接口或 Excel 行号机械罗列。
2. 现有文档提供业务场景基础，外部测试用例提供样本和补充要求，Playwright 脚本提供自动化证据。
3. 一个外部测试用例只映射到最贴近其主验证目标的矩阵场景；其他相关场景只能作为辅助/同类引用。
4. 覆盖状态以当前真实证据为准；没有执行证据或核心断言不足时，不标记为已覆盖。
5. mock、real、DB、文件、跨角色、跨系统证据按验证目标分层使用。
6. 涉及真实入库、唯一性、文件解析、权限隔离、跨页面回查、跨角色回查或后端最终状态的场景，不能只用 mock 结案。
7. 查询真实数据无命中时，如实记录查不到，不用 mock 命中结果掩盖真实样本缺失。
8. 已有 mock 用例若需要升级真实证据，优先修改原 test，不新增重复场景。
9. 外部测试用例中的不明确字段、接口、DB 表、账号权限、样本路径和业务规则必须列为待确认项。

## 3. 覆盖判定口径

| 覆盖类型 | 适用范围 | 可证明内容 | 不能证明内容 | 覆盖状态口径 |
|---|---|---|---|---|
| real | 真实页面、真实接口、真实数据、真实文件、真实角色权限 | 主链路、持久化结果、真实回显、真实状态变化 | 不易安全制造的极端异常 | 可作为已覆盖或部分覆盖证据 |
| mock | 前端分支、错误态、空态、按钮状态、弹窗、loading、toast、接口异常返回 | 页面交互、前端防护、受控异常呈现 | 真实入库、真实后端幂等、真实文件解析、真实权限隔离 | 只证明前端或受控分支 |
| real + mock | 主链路真实，少见异常或不可安全制造状态用 mock 补充 | 真实主流程加前端异常分支 | mock 不能替代真实业务结论 | 已覆盖时必须拆开说明各自证明什么 |
| DB 校验 | Excel 或业务规则明确要求查库，或页面无法证明最终状态 | 数据库字段、状态变更、落库记录、状态不变 | UI 展示是否正确 | 只有 DB 证据完成后才可升级对应核心结论 |
| 文件校验 | 上传、下载、导入、导出、模板、错误文件 | 文件名、格式、模板列、错误行、解析结果 | 后端最终业务状态，除非结合 DB/接口 | 作为文件链路证据 |
| 跨角色/跨系统 | 多角色回查、上下游同步、外部系统联动 | 数据流转、权限隔离、最终一致性 | 单页面局部交互 | 未确认入口、账号、SLA 前保持规划中或部分覆盖 |

矩阵要求：

1. 场景编号使用 `正常流01 / 边界流01 / 异常流01`，同类型内连续编号。
2. 场景名称必须表达主验证目标，不写成单纯按钮名称。
3. 操作步骤写用户动作和关键等待点，不写实现细节。
4. 核心断言写业务结果、UI/API/DB/文件证据，不只写页面出现文本。
5. 覆盖状态必须与缺口说明一致。

## 5. 测试用例映射表

仅当用户提供外部测试用例或仓库存在明确测试用例整理文档时生成。

| 场景编号 | 主映射测试用例 | 辅助/同类测试用例 | 映射理由 | 关联 Playwright 用例 |
|---|---|---|---|---|
| 正常流01 | Row / Case / 用例名 | Row / Case / 用例名 | 该用例的主验证目标与本场景一致 | `spec title` / 待新增 / 无 |

映射要求：

1. 主映射必须唯一。
2. 辅助/同类只能说明相关性，不能作为覆盖状态升级依据。
3. 无直接对应测试用例时写“无直接对应测试用例”。
4. 外部用例要求 DB、文件、跨系统或特殊字段时，应同步进入对应证据补充表。

### 9.5 批次执行与阻断处理

| 规则 | 说明 |
|---|---|
| 不阻塞独立场景 | 若某场景前置依赖未满足，不阻塞同批次其他独立场景继续落地 |
| 阻断场景处理 | 标记阻断条件后移至下一批次，覆盖状态保持不变 |
| 状态不预升级 | 不得因"预计下批次补上"而提前将 `部分覆盖` 升级为 `已覆盖` |
| 降级到 mock | 若真实验证短期不可行且场景本身允许，可临时降级为 mock 辅助，但必须在方案中标注"mock 临时降级，待真实环境就绪后升级" |

### 9.6 收口批：文档、状态和报告同步

1. 同步矩阵覆盖状态。
2. 同步测试用例映射关系。
3. 同步 README 或执行清单。
4. 同步审计报告入口和报告口径。
5. 清理过期计划项，避免已完成内容继续留在待办。

## 10. 完成标准

1. 矩阵完整覆盖当前已识别场景。
2. 外部测试用例主映射唯一，辅助映射不重复计数。
3. 覆盖状态诚实，缺口说明能解释为什么不是已覆盖。
4. 每个建议新增或修改的 Playwright 用例都有明确数据来源和核心断言。
5. 必须真实、DB、文件或跨系统的场景没有被 mock 替代结案。
6. 公共能力与批次计划能直接指导后续实现。
7. 待确认项全部列出，没有编造接口、字段、DB 表、账号、样本或业务规则。
